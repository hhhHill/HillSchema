# HillSchema 洞察优先版产品 Spec

## 1. 文档目的

本文档用于定义 HillSchema 第一版产品的目标、边界和整体方案。

这一版产品基于 `agentscope-dataagent` 做重度 fork，但不会沿用“聊天页作为主入口”的原始形态，而是改造成一个以“自动洞察流”为核心入口的数据 Agent 产品。

本文档回答的是以下问题：

- 第一版到底做什么
- 首页、详情页、对话、数据源、定时任务之间如何协同
- 哪些能力属于第一版
- 哪些能力明确后置

本文档不是开发排期文档，也不展开代码实现细节。

## 2. 一句话定义

HillSchema V1 是一个基于已登记数据源持续生成问题消息的数据 Agent 产品。

系统每 1 分钟自动执行一次洞察任务，把识别出的异常、趋势、归因或摘要类问题写入首页问题流。用户从首页点开某条问题后，可以先查看这条问题生成时的证据快照，再围绕该问题继续进行带上下文的对话分析。

## 3. 产品目标

第一版的核心目标只有一个：

先让系统持续发现问题，再让用户围绕问题继续分析。

围绕这个目标，V1 需要跑通如下闭环：

1. 接入多个已登记数据源。
2. 系统按固定频率自动执行洞察任务。
3. 洞察结果以“问题消息”的形式写入首页。
4. 用户点击问题后可以查看结论和证据快照。
5. 用户可以在该问题上下文中继续追问和分析。

## 4. 第一版产品形态

### 4.1 首页

首页是“自动洞察流”，不是传统聊天首页，也不是 BI 仪表盘。

首页展示的是系统已经生成好的问题消息，按时间倒序排列。每条消息至少包含：

- 生成时间
- 所属数据源
- 问题标题
- 一句话摘要
- 问题类型
- 当前状态

首页的目标不是让用户先输入，而是让用户先知道系统刚刚发现了什么。

### 4.2 问题详情

点击首页中的某条问题后，进入问题详情视图。

问题详情分成上下两层：

- 上半部分展示该问题生成当时的结论、证据和数据快照。
- 下半部分提供围绕该问题的上下文对话能力。

这样做的目的有两个：

- 让用户知道系统为什么会发出这条问题消息。
- 让用户可以从这条问题继续向下分析，而不是重新起一个泛化问题。

### 4.3 对话

第一版对话不是全局开放问答，而是“问题上下文对话”。

这意味着：

- 每条问题都有自己的上下文范围。
- 用户在问题详情页中的追问默认围绕当前问题展开。
- 当前问题切换后，对话上下文也随之切换。
- 模型优先回答与当前问题相关的时间、维度、归因、证据和后续分析问题。

它的产品定位不是“随便问数据库”，而是“围绕系统发现的问题继续深挖”。

### 4.4 Workspace

Workspace 保留，但不是第一版主入口。

第一版中，Workspace 的定位是辅助能力区，主要用于承接后续的技能、子 Agent、执行痕迹和工作目录等能力，不作为首页主体验重点。

## 5. 数据源范围

### 5.1 支持范围

第一版支持多个已登记数据源，优先面向 JDBC 类数据源，首批以 MySQL 和 PostgreSQL 为主。

### 5.2 权限边界

数据权限主要下沉到数据库账号本身控制，不在应用层重复建设复杂 ACL。

也就是说：

- 能否连上某个数据源，由数据源配置决定。
- 能否访问某些 schema、table、column，由数据库账号权限决定。
- 应用层只负责读取允许访问的数据，不重复模拟数据库权限体系。

### 5.3 轻量业务语义配置

虽然权限下沉到数据库，但为了让自动洞察稳定工作，每个已登记数据源仍需要提供一份轻量业务语义配置。

这份配置不是完整 schema 镜像，而是最小必要信息，主要包括：

- 电商核心表的真实映射
- 主时间列
- 关键关联关系
- 允许参与洞察的指标
- 不允许进入洞察与模型上下文的敏感字段

第一版业务域先收敛到统一的电商五张核心表：

- `orders`
- `order_items`
- `users`
- `products`
- `refunds`

因此，V1 的定义不是“任意数据库接入即自动理解”，而是“多个已登记数据源 + 统一的轻量电商语义映射”。

## 6. 自动洞察定义

### 6.1 自动洞察不是实时临时计算

自动洞察在第一版中更像系统主动发出的消息，而不是用户打开首页时临时跑出来的一屏实时查询结果。

系统行为应该是：

- 每 1 分钟触发一次洞察任务。
- 每次任务生成一批新的问题结果。
- 这些结果被写入系统内部存储。
- 首页读取的是已经生成好的问题消息流。

因此，首页承载的是“问题消息流”，不是“实时大查询结果页”。

### 6.2 固定调度方式

第一版中，洞察任务的调度策略固定为：

- 每 1 分钟执行一次

这个 1 分钟不是临时配置，也不是占位，而是第一版的明确产品行为。

### 6.3 洞察类型

第一版内部固定支持 4 类问题：

- 异常
- 趋势
- 归因
- 摘要

但首页不按类型做分区，而是统一作为问题流展示。

## 7. 洞察生成机制

第一版采用“本地计算 + 模型表达”的混合机制。

### 7.1 本地计算负责的部分

本地逻辑负责：

- 连接数据源
- 执行聚合查询
- 做窗口对比
- 识别异常或变化
- 生成归因候选
- 形成结构化问题候选

### 7.2 模型负责的部分

模型负责：

- 生成问题标题
- 生成一句话摘要
- 生成简洁结论
- 生成证据说明文案
- 生成建议追问方向

### 7.3 不采用的大模型方案

第一版不采用“把原始大表直接喂给模型，让模型自由找问题”的方案。

原因很明确：

- 成本高
- 稳定性差
- 可解释性弱
- 不利于后续做调度和治理

## 8. 问题消息的数据要求

每条问题消息至少需要沉淀以下信息：

- 问题唯一标识
- 所属数据源
- 生成时间
- 问题类型
- 问题标题
- 摘要
- 结论
- 状态
- 证据快照
- 对应的关键维度与指标
- 可继续追问的上下文基础

这里尤其重要的是“证据快照”。

因为用户点开问题时，数据库中的实时数据可能已经发生变化。如果不保留问题生成当时的证据，用户看到的内容就会和首页消息对不上。

所以第一版必须把问题生成当时的关键证据一起保存下来，而不是只保存一个问题标题。

## 9. 第一版范围内保留但降级的能力

基于 `agentscope-dataagent` 原有能力，第一版仍然保留以下基础设施：

- Chat 能力
- Workspace 能力
- Skill / Subagent 扩展能力
- 运行时与工具注册能力
- Session 能力

但这些能力在第一版中的角色是“支撑层”，不是“主入口产品体验”。

## 10. 第一版明确不做的内容

为了避免范围失控，第一版明确不做以下事情：

- 不做跨多个数据源的联合洞察
- 不做任意 schema 自动理解
- 不做用户自定义刷新周期
- 不做用户自定义分析时间区间
- 不做模型自动推荐时间区间
- 不把首页做成可视化大盘
- 不把 marketplace、审批流、多渠道接入作为主体验重点
- 不把全局开放式数据库问答作为第一版主路径

## 11. 已确认的后续 TODO

以下内容不是第一版交付范围，但需要明确记录，后续迭代处理：

1. 让用户自己配置时间区间，或者由模型推荐更合理的分析时间区间。
2. 先由本地计算完成异常识别，再让模型负责表达，不把原始大表直接送入模型上下文。

其中第二条在第一版已经按方向约束，但后续还需要做得更彻底、更体系化。

## 12. 成功标准

如果第一版成功，用户应该能在一个最小演示闭环中体验到以下过程：

1. 系统已经接入多个已登记数据源。
2. 系统每分钟自动生成新的问题消息。
3. 用户进入首页时，先看到问题流，而不是空白聊天页。
4. 用户点开任意问题后，可以看到生成当时的证据快照。
5. 用户可以围绕该问题继续追问，并得到带上下文的分析结果。

只要这个闭环稳定成立，第一版目标就达成了。
